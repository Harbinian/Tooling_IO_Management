# 8D 问题解决协议 / 8D Problem Solving Protocol

仅用于调试或回归问题。

---

## 原则 / Principles

- **拒绝临时补丁**: 必须深挖根因，从代码、测试、流程三个维度预防复发
- **防退化红线**: 禁止"婴儿与洗澡水同倒"，必须死守旧有的核心业务边界
- **挂起节点**: D3、D5、D6 节点完成后必须通知 `reviewer` 审核，通过后才能继续
- **思想钢印 - 证据链先行**: D2 问题描述必须基于**可观测证据**（日志、错误信息、堆栈、复现步骤），禁止在证据不足时进行根因推测。根因是 D4 阶段的产物，D2 的任务只有一条：**忠实记录现象，不解释原因**。

---

## 流程 / Steps

| 阶段 | 内容 | 审核 |
|------|------|------|
| D1 | 团队分工 (Reviewer, Coder, Architect) | - |
| D2 | 问题描述 (证据链 + 5W2H，禁止根因推测) | - |
| D3 | 临时遏制措施 (Containment) | G4-1 结构检测通过 → **评分审核** |
| D4 | 根因分析 (5 Whys) | - |
| D5 | 永久对策 + 防退化宣誓 | G4-1 结构检测通过 → **评分审核** |
| D6 | 实施验证 (Implementation) | G4-1 结构检测通过 → **评分审核** |
| D7 | 预防复发 (Prevention) | - |
| D8 | 归档复盘 (Documentation) | - |

---

## 团队协作 / Team Protocol

**Reviewer Agent** 职责：
- 接收挂起通知，审查 D3/D5/D6 的输出
- 验证：根本原因是否充分、方案是否防退化、代码是否完整
- **使用评分制进行量化评估**，回复评分结果

**工作流**：
```
D3/D5/D6 完成 → G4-1 结构检测（CI 自动，阻断不通过）
              → SendMessage(to: "reviewer", type: "plan_approval_request")
              → reviewer 评分审核
              → 收到全部维度达标后继续
```

> **G4-1 定位**：CI 自动执行的文档结构预检，不替代 reviewer 评分。G4-1 通过是进入 reviewer 评分的前提条件。

### D2 证据清单格式 / Evidence Chain Template

**必须包含以下证据（至少三项），缺一不可进入 D3：**

| 证据类型 | 内容 | 来源 |
|----------|------|------|
| 错误日志 | 完整的 error/exception 输出 | 服务器日志、控制台 |
| 堆栈跟踪 | 异常堆栈或调用链路 | Stack trace、Debug 日志 |
| 复现步骤 | 从初始状态到 bug 出现的完整操作序列 | 测试记录、用户报告 |
| 影响范围 | 受影响的用户数、功能模块、订单范围 | 监控、用户反馈 |
| 对比基准 | 正常行为 vs 异常行为的差异点 | 测试环境、Prod 对比 |

**D2 阶段禁止出现的内容**：
- ❌ "可能是 XXX 导致的"
- ❌ "根据经验判断是 YYY"
- ❌ "根因是 ZZZ"

**D2 阶段的唯一任务**：用证据描述**是什么**（What），而不是**为什么**（Why）。Why 留给 D4。

---

### D2 信息缺口识别 / Information Gap Detection

执行者发现以下情况时，**必须暂停并主动询问**，不得擅自假设：

| 场景 | 必问项 |
|------|--------|
| 账号/权限类 bug | 报告人的**用户名**、**角色**、**所属组织**、**触发操作的完整步骤** |
| UI 行为异常 | 浏览器/环境、前端控制台错误信息、是否在多个账号下复现 |
| 状态机流转异常 | 订单号、订单类型（出库/入库）、当前状态、操作时间线 |
| 通知类 bug | 通知发送方/接收方账号、通知渠道（飞书/企微）|

**追问模板**：
```
【D2 信息缺口】以下信息未在证据链中找到，请补充：

1. 报告人账号：[username] / 角色：[role] / 组织：[org]
2. 触发操作的完整步骤：
   Step 1: ...
   Step 2: ...
   Step 3: ...
3. 复现时间：[timestamp]
4. 错误日志/控制台截图：[附件]

在收到以上信息前，禁止进入 D3。
```

**什么时候可以跳过追问**：
- 证据链已完整（至少 3 项证据齐全）
- 复现路径清晰、可在测试环境独立验证

## 评分制审核 / Scoring Review

在 D3/D5/D6 节点，reviewer 必须使用评分制进行量化评估。

### 评估维度

| 维度 | 权重 | 评分标准 (0-1) |
|------|------|----------------|
| root_cause_depth | 0.3 | 根因分析是否穿透到真正原因（而非表面现象） |
| solution_completeness | 0.3 | 方案是否完整覆盖所有影响点 |
| code_quality | 0.2 | 代码是否符合项目规范（命名/编码/错误处理） |
| test_coverage | 0.2 | 是否有对应测试验证修复有效 |

### 评分输出模板

reviewer 必须输出以下格式：

```markdown
## 评分结果

| 维度 | 得分 | 最低要求 | 结论 |
|------|------|----------|------|
| root_cause_depth | 0.8 | ≥0.24 | ✅ PASS |
| solution_completeness | 0.7 | ≥0.24 | 🔶 FAIL |
| code_quality | 0.9 | ≥0.16 | ✅ PASS |
| test_coverage | 0.6 | =0.20 | ❌ FAIL |

**加权总分**: 0.75/1.0

**结论**:
- 全部维度达标 → ✅ APPROVE
- 任意维度不达标 → ❌ REJECT (必须重新做)
```

### 各维度最低门槛

| 维度 | 满分 | 最低门槛 | 说明 |
|------|------|----------|------|
| root_cause_depth | 0.3 | ≥0.24 | 根因分析必须达到满分的 80% |
| solution_completeness | 0.3 | ≥0.24 | 方案完整性必须达到满分的 80% |
| code_quality | 0.2 | ≥0.16 | 代码质量必须达到满分的 80% |
| test_coverage | 0.2 | ≥0.16 | 新增代码必须有测试验证，详见评分参考 |

**任一维度不达标则驳回。**

### 评分标准细则

**root_cause_depth** (满分 0.3，门槛 ≥0.24):
- 1.0: 找到真正根因，触及系统级问题
- 0.8: 找到直接原因，但未完全穿透到系统层
- 0.6: 只找到触发因素，未触及根本原因
- < 0.24: 仅处理了症状

**solution_completeness** (满分 0.3，门槛 ≥0.24):
- 1.0: 所有影响路径都有对应修复
- 0.8: 主要路径已覆盖，边界情况遗漏
- 0.6: 修复了部分问题，仍有遗漏
- < 0.24: 修复不完整或引入新问题

**code_quality** (满分 0.2，门槛 ≥0.16):
- 1.0: 符合所有项目规范，无警告
- 0.8: 有少量格式问题但不影响功能
- 0.6: 代码可工作但质量较差
- < 0.16: 代码质量差，难以维护

**test_coverage** (满分 0.2，门槛 ≥0.16):
- 1.0: 有完整单元测试 + 边界测试
- 0.8: 有主要路径测试，边界测试不足
- 0.6: 有简单测试，覆盖率不足
- < 0.16: 无测试或测试无效

> **test_coverage 评分参考**：
> - 1.0 分：新增代码覆盖率 ≥ 80% 且覆盖所有修改路径
> - 0.8 分：新增代码覆盖率 ≥ 60%
> - 0.6 分：仅有手动验证记录，无自动化测试
> - 0 分：无任何测试

---

## 要求 / Requirements

每个 bug 修复必须包含：

- 根本原因解释 / root cause explanation
- 测试用例 / test case
- 回归预防 / regression prevention

**禁止使用临时补丁** / Temporary patches are forbidden.
