# 提示词任务编号约定 / Prompt Task Numbering Convention

---

## 目的 / Purpose

定义提示词文件的编号规则和归档命名格式。

---

## 编号体系 / Numbering System

**统一使用5位类型编号，贯穿 active 和 archive 全流程：**

### active 目录格式

```
<类型编号>_<描述>.md
```

| 范围 | 类型 | 示例 |
|------|------|------|
| 00001-09999 | 功能任务 | `00017_order_list_ui_migration.md` |
| 10101-19999 | Bug修复任务 | `10101_bug_tool_search_routing.md` |
| 20101-29999 | 重构任务 | `20101_refactor_split_tool_io_service.md` |
| 30101-39999 | 测试任务 | `30101_workflow_state_machine_tests.md` |

### archive 目录格式

```
✅_<执行顺序号>_<类型编号>_<描述>_done.md
```

| 字段 | 说明 |
|------|------|
| `✅_` | 完成标记 |
| 执行顺序号 | 00001-99999，按执行顺序自增 |
| 类型编号 | 与 active 目录一致 |
| `_done.md` | 固定后缀 |

---

## 类型编号范围 / Type Number Ranges

| 类型编号范围 | 类型 | 说明 |
|-------------|------|------|
| `00001-09999` | 功能任务 | 功能开发、UI迁移等 |
| `10101-19999` | Bug修复任务 | Bug修复 |
| `20101-29999` | 重构任务 | 代码重构 |
| `30101-39999` | 测试任务 | 自动化测试 |

---

## 禁止事项 / Prohibitions

1. **禁止重复执行顺序号**: 每个归档文件必须有唯一的执行顺序号
2. **禁止缺少后缀**: 所有归档文件必须包含 `_done.md` 后缀
3. **禁止 🔶 前缀进入归档**: 执行后前缀为 🔶 的提示词不得移入 archive 目录，必须先修正为 ✅ 前缀
4. **禁止跳过审核节点归档**: Bug 修复任务如缺少 D3/D5/D6 reviewer 评分记录，禁止归档
5. **禁止无 E2E 验证归档**: 功能/重构任务如无 tester E2E 验证 pass 记录，禁止归档

---

## 归档前置条件 / Archive Prerequisites

### Bug 修复 (10101-19999) — 必须验证

| 条件 | 验证方式 |
|------|---------|
| D3 reviewer 评分全部维度达标 | 评分记录引用 |
| D5 reviewer 评分全部维度达标 | 评分记录引用 |
| D6 reviewer 评分全部维度达标 | 评分记录引用 |
| 所有 Completion Criteria 满足 | 检查清单 |

**各维度最低门槛**：

| 维度 | 满分 | 最低门槛 |
|------|------|----------|
| root_cause_depth | 0.3 | ≥0.24 |
| solution_completeness | 0.3 | ≥0.24 |
| code_quality | 0.2 | ≥0.16 |
| test_coverage | 0.2 | =0.20（必须满分） |

### 功能开发 (00001-09999) / 重构 (20101-29999) — 必须验证

| 条件 | 验证方式 |
|------|---------|
| Phase 4 E2E 验证 pass | tester 回复记录 |
| 所有 Completion Criteria 满足 | 检查清单 |

### 测试任务 (30101-39999) — 必须验证

| 条件 | 验证方式 |
|------|---------|
| 所有 P0 测试用例通过 | pytest 输出 |
| 覆盖率达标 | 覆盖率报告 |
| 所有 Completion Criteria 满足 | 检查清单 |

---

### 🔶 前缀处理规则

`🔶` 前缀表示"部分完成，存在违规"。处理方式：

1. **如果违规可修复**: 修正后重新执行，完成后将 🔶 改为 ✅ 再归档
2. **如果违规不可修复**: 拒绝归档，删除该提示词文件
3. **绝对禁止**: 将 🔶 文件直接归档

## 执行者分配规则 / Executor Assignment Rules

### 统一执行规则 / Unified Execution Rule

| 任务类型 | 正常执行者 | 简化任务 |
|---------|---------|---------|
| 功能任务 (00001-09999) | Gemini (设计), Codex (实现) | Claude Code |
| Bug 修复 (10101-19999) | Codex | Claude Code |
| 重构任务 (20101-29999) | Codex | Claude Code |
| 测试任务 (30101-39999) | Claude Code | - |

**简化任务**：参数问题、调用错误、签名修正、文档更新、单点修复 → 直接由 Claude Code 执行

### 简化任务标准 / Simple Task Criteria

以下任务视为"简化任务"，由 **Claude Code 直接执行**，不生成提示词：

1. **参数问题**: 类型不匹配、参数传递错误
2. **调用错误**: 函数/方法调用参数提取不正确
3. **签名修正**: 函数签名与实现不一致
4. **文档更新**: docstring、注释同步
5. **单点修复**: 不涉及架构变更的单文件/单函数修改

### 恶性 Bug 定义 / Malignant Bug Definition

- **P0 级别**: 系统无法运行、数据损坏风险
- **P1 级别**: 核心功能损坏、API 不可用、工作流阻塞

## 计数器文件 / Counter File

为避免频繁扫描 archive 目录获取编号，系统使用 `promptsRec/.sequence` 文件维护编号计数器。

### 文件格式

```
# 功能开发 (00001-09999)
feature_next=00019

# Bug修复 (10101-19999)
bugfix_next=10178

# 重构 (20101-29999)
refactor_next=20101

# 测试 (30101-39999)
test_next=30101

# 执行顺序号（归档用）
exec_next=00182
```

### 计数器说明

| 计数器 | 用途 | 范围 |
|--------|------|------|
| feature_next | 功能开发提示词编号 | 00001-09999 |
| bugfix_next | Bug修复提示词编号 | 10101-19999 |
| refactor_next | 重构提示词编号 | 20101-29999 |
| test_next | 测试提示词编号 | 30101-39999 |
| exec_next | 执行顺序号（归档用） | 00001-99999 |

### 使用规则

1. **生成新提示词时**: 读取对应类型的计数器值作为编号，然后原子递增
2. **归档时**: 读取 `exec_next` 作为执行顺序号，然后原子递增
3. **禁止扫描 archive**: 严禁通过扫描 archive 目录来确定编号
