# 提示词有效性审查报告

## 1. 审查范围

本次审查对象为以下 4 份活动提示词：

1. `10190_fix_inspection_api_500.md`
2. `10193_fix_login_api_401_inconsistency.md`
3. `10194_fix_rbac_permission_mismatch.md`
4. `20113_simplify_e2e_infrastructure.md`

审查目标是判断其是否满足仓库当前 `RUNPROMPT` 执行要求，是否具备正式执行、验证、归档的基本有效性。

## 2. 审查依据

本次审查主要依据以下规则与事实：

1. Bug 修复任务必须符合 8D 协议，参考 `.claude/rules/02_debug.md`
2. 重构任务必须符合 ADP 四阶段协议，参考 `.claude/rules/01_workflow.md`
3. 提示词编号、执行器、归档前提需符合 `.claude/rules/05_task_convention.md`
4. 当前编号状态以 `promptsRec/.sequence` 为准
5. 权威文档和引用文件必须在仓库中真实存在，不能依赖失效路径

## 3. 总体结论

总体判定如下：

| 提示词 | 类型 | 当前结论 | 是否可直接执行 |
|------|------|------|------|
| `10190` | Bug Fix | 部分有效，需整改 | 否 |
| `10193` | Bug Fix | 无效 | 否 |
| `10194` | Bug Fix | 部分有效，需整改 | 否 |
| `20113` | Refactor | 无效 | 否 |

结论说明：

- `10190` 内容相对完整，接近可执行，但仍存在流程缺口和引用问题
- `10193` 任务边界与规则冲突较大，不适合作为正式提示词
- `10194` 有任务方向，但完成标准和协议结构不合格
- `20113` 同时存在流程协议、依赖、文档落点和编号状态问题，当前不成立

## 4. 分项审查结果

### 4.1 `10190_fix_inspection_api_500.md`

判定：部分有效，需整改

优点：

1. 编号位于 `10101-19999`，类型区间合法
2. P1 Bug 由 `Claude Code` 执行，符合恶性 Bug 分配规则
3. 问题背景、影响接口、相关代码位置描述较完整
4. 明确采用 8D 结构，整体形态最接近正式提示词

主要问题：

1. 引用了 `docs/PROMPT_TASK_CONVENTION.md`，但仓库实际存在的是 `docs/archive/PROMPT_TASK_CONVENTION.md`，路径失真
2. 修复方案写得过于具体，已经接近“指定实现”，弱化了真实排障空间
3. D3、D5、D6 仍然只是 reviewer 占位内容，不能视为满足归档前提
4. 提示词中对 `reschedule_task()` 的问题判断带有推测性，证据链还不够完整

审查意见：

- 可保留为工作草稿
- 不应直接作为“已达标正式 bug prompt”执行和归档
- 建议补齐真实 reviewer 节点、修正文档引用，并减少对实现细节的预设

### 4.2 `10193_fix_login_api_401_inconsistency.md`

判定：无效

主要问题：

1. 该任务本质更接近“单点测试修复/调用对齐”，按仓库规则更像应由 `Claude Code` 直接处理的简化任务，不宜单独生成正式 bug prompt
2. `Stage: 1` 的头部元数据与同仓库编号体系不一致，规范性不足
3. 提示词正文承认根因可能来自字段名、session、headers、CSRF 或后端认证，但约束又写死“只能修改 `test_runner/playwright_e2e.py`，不能修改后端”，边界互相冲突
4. 缺少 8D 结构，不符合 bug fix 提示词最低要求
5. 完成标准更偏“现象修复”，缺少根因、回归预防、reviewer 节点要求

审查意见：

- 当前版本不应进入正式 `RUNPROMPT`
- 如需保留，应重写为：
- 简化任务，直接执行，不走完整 prompt 编排
- 或者重写为完整 8D bug prompt，允许根据真实根因决定改测试还是改后端

### 4.3 `10194_fix_rbac_permission_mismatch.md`

判定：部分有效，需整改

优点：

1. 编号区间合法
2. 执行器 `Codex` 与普通 Bug 类型基本匹配
3. 引用了 `docs/RBAC_PERMISSION_MATRIX.md` 和 `docs/API_SPEC.md`，方向正确
4. 已明确存在权限期望与实际不一致的问题

主要问题：

1. 缺少 8D 结构，不符合 bug 修复任务协议
2. “通过率 ≥ 80%”作为完成标准不合格，这等于允许已知失败保留，与 bug 修复和防回归要求冲突
3. “添加调试输出”被写为必做项，但这属于排查手段，不应直接成为正式交付标准
4. `Stage: 1` 元数据仍不规范
5. 提示词没有明确界定：若是文档错、测试错、实现错，谁是权威源、如何判定变更顺序

审查意见：

- 可作为初始排查草稿
- 不可作为正式可归档 bug prompt
- 必须重写完成标准，至少应要求：
- 根因明确
- 权限矩阵与实现一致
- 受影响用例全部通过
- 无新增回归

### 4.4 `20113_simplify_e2e_infrastructure.md`

判定：无效

主要问题：

1. 这是重构任务，但正文未遵守 ADP 的 `PRD -> Data -> Architecture -> Execution` 四阶段要求，只写了执行型清单
2. 依赖 `10193_fix_login_api_401_inconsistency`，而 `10193` 当前本身无效，导致依赖关系不成立
3. 完成标准要求更新 `CLAUDE.md`，但仓库内未检出该文件，属于落点失效
4. 提示词引用的 `AI_PIPELINE.md`、`README_AI_SYSTEM.md`、`PROMPT_TASK_CONVENTION.md` 当前可见位置在 `docs/archive/`，说明引用体系与当前仓库权威文档不一致
5. `promptsRec/.sequence` 仍然显示 `refactor_next=20113`，说明该编号未完成正常消耗，流程状态异常
6. 提示词要求“归档历史报告、清理数据库”等，但没有明确边界和风险控制，对重构任务而言过宽

审查意见：

- 当前版本不应执行
- 需要先重新定义：
- 目标是“测试策略调整”还是“测试基础设施重构”
- 哪些是文档治理，哪些是代码治理
- 哪些测试资产允许归档或清理
- 同时必须先修正编号计数器状态

## 5. 共性问题归纳

本次 4 份提示词暴露出以下共性问题：

1. 文档引用与仓库现实脱节
- 多处引用 `docs/PROMPT_TASK_CONVENTION.md` 等路径，但当前可见文件位于 `docs/archive/`
- 这会导致执行者误判权威文档位置

2. 流程协议未真正落地
- Bug 任务没有完整 8D
- Refactor 任务没有完整 ADP
- 说明提示词编写停留在“任务描述”层，而不是“可执行流程资产”层

3. 验收标准偏松或偏虚
- 典型问题是 `10194` 的“通过率 ≥ 80%”
- 正式 bug 修复不应允许残留已知失败作为达标条件

4. 任务边界与根因假设冲突
- `10193` 最明显，既承认根因未明，又提前限制只能改测试文件
- 这会把提示词从“排障任务”扭曲成“定向修补任务”

5. 编号和流程状态治理不足
- `20113` 与 `.sequence` 状态冲突，说明提示词生成和计数器维护不同步

## 6. 风险评估

如果直接按当前版本执行，主要风险如下：

1. 执行者依据失效文档路径工作，导致引用错误的流程规范
2. 在根因未确认时被提示词强行限制改动范围，形成错误修复
3. 任务完成后仍无法满足归档要求，需要二次返工
4. 编号状态冲突继续扩大，后续提示词存在撞号风险
5. 测试或权限问题被以“部分通过”方式错误关闭，形成长期回归隐患

## 7. 整改建议

建议按优先级整改：

1. 先修正提示词模板和引用路径
- 统一只引用仓库当前真实存在的权威文档
- 清理失效的 `docs/PROMPT_TASK_CONVENTION.md`、`CLAUDE.md` 等引用

2. 重写 `10193` 和 `20113`
- `10193` 要么降级为简化任务直接执行，要么补齐完整 8D
- `20113` 必须按 ADP 重写，并先修正依赖链和编号状态

3. 修正 `10194` 的完成标准
- 去掉“通过率 ≥ 80%”这类宽松条件
- 改为受影响权限场景全部验证通过，且无新增回归

4. 补齐 `10190` 的正式审查节点
- 若要归档，必须补完 D3/D5/D6 reviewer 评分结果
- 同时把文档引用改为当前有效路径

5. 检查 `promptsRec/.sequence`
- 重点核对 `refactor_next=20113` 是否应前移
- 避免后续重构提示词重复占号

## 8. 最终审查结论

本次审查结论为：

1. `10190` 可以整改后继续使用
2. `10194` 可以整改后继续使用
3. `10193` 应重写或降级为简化任务
4. `20113` 应整体重写，并先修复流程状态问题

结论性意见：

这 4 份提示词目前整体上更像“工作草稿”，而不是满足仓库流程要求的正式执行资产。若用于正式 `RUNPROMPT`，建议先完成整改，再进入锁定、执行、报告与归档流程。
