# `.skills` 与 `.claude` 冲突修复建议

## 1. 目标

本建议文档用于指导如何修复 `E:\CA001\Tooling_IO_Management\.skills` 与 `E:\CA001\Tooling_IO_Management\.claude` 之间的规则冲突，使技能执行重新回到仓库规范真源之下。

目标不是继续增强 `.skills` 的“规则定义能力”，而是做以下三件事：

1. 明确 `.claude` 是唯一规则真源
2. 让 `.skills` 只负责触发、编排、校验和产出
3. 移除技能层重复定义、冲突定义和错误判断逻辑

---

## 2. 修复总原则

### 2.1 真源唯一原则

所有流程规则、任务分类、执行器规则、归档前置条件、编号规则，统一以 `.claude/rules/` 为准。

`.skills` 中不得重新定义以下内容：

- ADP 四阶段正文
- 8D 协议正文
- HOTFIX SOP 正文
- 执行器分配标准
- 简化任务标准
- 编号范围和编号生成规则
- reviewer/tester 归档前置条件

`.skills` 只能写：

- 何时调用哪条规则
- 调用前要检查什么
- 调用后要产出什么
- 出现不满足条件时如何中止

---

### 2.2 技能降权原则

技能文档应从“规则定义层”降为“流程适配层”。

应删除或改写所有会被模型理解为“主规则”的句式，例如：

- 始终由某执行器执行
- 若 prompt 另有规定则以 prompt 为准
- 技能内自定义的评分规则
- 技能内自定义的归档图标规则

这些内容如果已经在 `.claude` 中定义，技能文件只能引用，不得再描述第二遍。

---

### 2.3 编号优先于命名原则

所有 prompt 类型识别应优先按编号区间判断，不得依赖 `_bug_`、`_fix_`、`_refactor_` 等文件名片段。

推荐统一识别规则：

- `00001-09999` → Feature
- `10101-19999` → Bug Fix
- `20101-29999` → Refactoring
- `30101-39999` → Testing

文件名语义仅作辅助展示，不得作为协议路由条件。

---

### 2.4 `.sequence` 唯一计数器原则

编号分配和执行顺序号只能来源于 `promptsRec/.sequence`。

允许读取 `archive/` 的场景应收缩到最低，只限：

- 人工查看历史
- 非关键展示

不得将 `archive/` 用于：

- 推断新编号
- 判断任务类型
- 自动拼装 follow-up 主档
- 替代 `.sequence` 或主流程状态

---

### 2.5 锁一致性原则

只要进入 `RUNPROMPT` 执行链路，就必须先创建 `.lock`。

不允许区分：

- `Executor = Claude Code` 不上锁
- 其他执行器才上锁

统一规则应为：

1. 选中 active prompt
2. 先创建 lock
3. 再开始读取和执行
4. 结束后按结果移除 lock

---

## 3. 技能级修复建议

## 3.1 `prompt-task-runner/SKILL.md`

这是最高优先级整改文件。

### 必修项

1. 删除“以提示词中的指示为准”条款
2. 删除按 `*_bug_*.md` 识别 bug prompt 的逻辑
3. 改为按编号区间识别任务类型
4. 删除对 ADP/8D/HOTFIX 的正文复写，只保留引用
5. 删除或收缩扫描 `archive/` 做 follow-up merge 的逻辑
6. 保留 `.sequence` 作为唯一编号和执行顺序来源
7. 明确无论执行器是谁，都必须先创建 `.lock`

### 推荐写法

建议把该技能收敛成以下结构：

1. 读取 active prompt
2. 解析编号
3. 根据编号区间选择对应 `.claude/rules/*.md`
4. 检查依赖、锁、前置条件
5. 执行
6. 写报告
7. 在满足 `.claude` 归档前提时归档
8. 清理 lock

### 应删掉的内容类型

- 二次定义执行器规则
- 二次定义 bug 审核规则
- 二次定义归档图标策略
- 用文件名片段判断任务类型

---

## 3.2 `auto-task-generator/SKILL.md`

这是第二优先级整改文件。

### 必修项

1. 删除与 `.claude/rules/05_task_convention.md` 冲突的执行器表述
2. 删除对“简化任务”的扩写定义
3. 保留“如何分类”的说明，但分类后必须回落到 `.claude` 真源判断
4. 增加“生成前校验”

### 生成前必须校验的内容

1. 目标文档是否真实存在
2. 引用规则文件是否真实存在
3. 目标代码文件是否真实存在
4. 编号是否来自 `.sequence`
5. 执行器是否与 `.claude` 一致
6. 如果是简化任务，是否根本不该生成 prompt

### 推荐写法

技能文件不要再写：

- “普通 bug 由谁执行”
- “重构始终由谁执行”
- “不涉及架构变更的 bug 都算简化任务”

技能文件只需写：

- 发现问题
- 初步分类
- 调用 `.claude/rules/05_task_convention.md` 校验
- 若通过则生成 prompt
- 若属于简化任务则拒绝生成并交由直接执行链路

---

## 3.3 `self-healing-dev-loop/SKILL.md`

这是第三优先级整改文件。

### 必修项

1. 删除“Executor = Claude Code 时不创建 lock”的规则
2. 删除或弱化内部重复定义的 reviewer/tester 规则正文
3. 保留 orchestration 逻辑，但所有判断落回 `.claude`
4. 明确它只是协调器，不是规则真源

### 推荐职责

该技能应只负责：

1. 发现问题
2. 触发任务生成
3. 触发 RUNPROMPT
4. 跟踪状态
5. 更新面板

它不应负责：

1. 改写执行器规则
2. 改写 lock 规则
3. 改写归档前置条件
4. 改写审查门槛

---

## 3.4 `prompt-generator/SKILL.md`

### 必修项

1. 删除大段 ADP/8D/HOTFIX 正文拷贝
2. 把“协议正文”改为“协议引用”
3. 增加真实落点校验
4. 增加“文档存在性检查失败则拒绝生成”的规则

### 推荐的生成前硬校验

1. 引用的 `docs/*.md` 是否存在
2. 引用的 `backend/`、`frontend/` 文件是否存在
3. 任务类型是否与编号区间一致
4. 执行器是否与 `.claude` 一致
5. 该任务是否属于不应生成 prompt 的简化任务

---

## 4. 规则与技能的正确分层建议

建议将规则分层固定为以下结构：

### 层 1：规则真源层

位置：

- `.claude/rules/`

负责：

- 定义所有标准
- 定义审核门槛
- 定义执行器映射
- 定义归档前提
- 定义编号与计数器规则

### 层 2：技能编排层

位置：

- `.skills/`

负责：

- 检测使用场景
- 选择调用哪条规则
- 串联步骤
- 执行前做存在性和依赖校验
- 执行后生成报告

### 层 3：提示词实例层

位置：

- `promptsRec/active/`
- `promptsRec/archive/`

负责：

- 承载具体任务实例
- 不得反向定义仓库级规则

---

## 5. 建议落地的硬规则

以下建议直接作为后续技能修订准则：

### 规则 1

`.skills` 不得复写 `.claude/rules` 的协议正文，只能引用文件路径与适用场景。

### 规则 2

任务类型识别只按编号区间，不按文件名片段。

### 规则 3

执行器分配只按 `.claude/rules/05_task_convention.md`，技能层不得再维护一套执行器映射表。

### 规则 4

简化任务标准只按 `.claude/rules/05_task_convention.md`，技能层不得扩写。

### 规则 5

任何进入 RUNPROMPT 链路的任务，执行前必须创建 `.lock`。

### 规则 6

编号和执行顺序号只允许从 `promptsRec/.sequence` 读取。

### 规则 7

生成 prompt 前必须校验：

- 引用路径存在
- 目标文件存在
- 执行器匹配
- 任务分类匹配
- 是否属于不应生成 prompt 的简化任务

### 规则 8

任何技能文件中不得出现“若 prompt 中另有规定则以 prompt 为准”这类反向覆盖规则条款。

### 规则 9

归档判断只按 `.claude` 归档前置条件执行，不允许技能层发明新的图标逻辑或替代逻辑。

### 规则 10

如 `.claude/rules` 更新，`.skills` 只能检查引用路径是否仍有效，不应同步复制更新正文。

---

## 6. 推荐整改顺序

建议按以下顺序整改：

1. ` .skills/prompt-task-runner/SKILL.md `
2. ` .skills/auto-task-generator/SKILL.md `
3. ` .skills/self-healing-dev-loop/SKILL.md `
4. ` .skills/prompt-generator/SKILL.md `

原因：

1. `prompt-task-runner` 直接影响 prompt 执行与归档，风险最大
2. `auto-task-generator` 直接影响新 prompt 的质量和合规性
3. `self-healing-dev-loop` 影响自动编排和并发控制
4. `prompt-generator` 影响模板和提示词结构质量

---

## 7. 推荐验收标准

修复完成后，建议用以下标准验收：

1. `.skills` 中不再出现独立执行器规则表
2. `.skills` 中不再出现 ADP/8D/HOTFIX 正文大段复写
3. bug、feature、refactor、testing 类型均按编号区间识别
4. Claude Code 进入 RUNPROMPT 时也会创建 `.lock`
5. 生成新 prompt 前会拒绝失效文档引用
6. 简化任务不会再错误生成正式 prompt
7. 任意技能文档都不能覆盖 `.claude` 规则

---

## 8. 结论

修复方向应当是“收权”，不是“补充更多技能说明”。

具体来说：

1. `.claude` 继续做唯一真源
2. `.skills` 删除重复规则正文
3. `.skills` 只保留场景触发、引用、前置校验、执行编排和产出说明

这样才能从结构上防止后续再次发生规则漂移。
