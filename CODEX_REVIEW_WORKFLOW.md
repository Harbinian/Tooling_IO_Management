# Codex 审查后标准操作清单

适用场景：你在 `https://chatgpt.com/codex` 完成一轮代码审查后，需要决定下一步是修复、提交、同步远端，还是继续下一轮审查。

---

## 0. 名词解释

这一节先把后文会反复出现的术语说明白。你以后看到这些词时，按这里的定义理解。

### 0.1 仓库 / Repository

仓库就是当前项目的 git 管理目录，也就是项目根目录 [`E:\CA001\Tooling_IO_Management`](E:\CA001\Tooling_IO_Management)。

它同时包含两类东西：

- 业务文件：代码、文档、配置、测试、提示词、日志等
- git 元数据：提交历史、分支、远端跟踪关系、暂存状态等

后文说“检查仓库状态”，本质上是在看文件变化和 git 记录之间的关系。

### 0.2 工作区 / Working Tree

工作区就是你磁盘上当前真实存在、可编辑的文件状态。

例如：

- 你改了一个 `.py` 文件但还没提交
- 你新增了一个测试文件
- 你删除了一个 `.md` 文件

这些都属于工作区变化。

“工作区干净”通常表示：

- 没有未提交的文件修改
- 没有未处理的新增、删除、重命名
- 没有悬而未决的冲突

“工作区不干净”表示你本地还有现场没有处理完。这时直接 `pull` 风险很高。

### 0.3 暂存区 / Index / Staging Area

暂存区是 git 用来准备下一次提交的中间层。

你可以理解为：

- 工作区：你正在编辑的真实文件
- 暂存区：你准备这次提交要带上的变更清单
- 提交历史：已经正式写进 git 的版本记录

最常见动作：

```powershell
git add <file>
```

这个命令不是提交，只是把工作区里的某些改动放进暂存区。

### 0.4 提交 / Commit

提交是一次正式的版本记录。

一次提交通常代表：

- 一组经过筛选的变更
- 一个明确的目的
- 一个可回溯的历史节点

在代码审查流程里，提交最好满足两个条件：

- 语义清晰：别人一看 commit message 就知道你修了什么
- 范围收敛：不要把无关改动、测试产物、流程噪音混进去

### 0.5 分支 / Branch

分支是提交历史上的一个可移动指针。

常见理解：

- `main`：主线分支
- `feature/...`：功能开发分支
- `fix/...`：问题修复分支

建议：

- 小修复也尽量不要长期直接堆在 `main`
- 审查发现问题后，优先在独立修复分支上处理

### 0.6 当前分支 / HEAD

当前分支就是你此刻所在的位置。

查看方式：

```powershell
git rev-parse --abbrev-ref HEAD
```

如果输出是 `main`，说明你当前就在主线分支上操作。此时任何修改都更需要谨慎，因为主线通常要保持可同步、可审查、可回滚。

### 0.7 远端 / Remote

远端就是你本地仓库对应的线上仓库地址。

最常见的是：

- `origin`

查看方式：

```powershell
git remote -v
```

这里的 `origin` 不是特殊关键字，只是默认名字。它通常指向 GitHub 上的那个仓库。

### 0.8 远端跟踪分支 / Remote-Tracking Branch

例如：

- `origin/main`

它表示“你本地所知道的远端 `main` 状态快照”，不是实时远端本身。

这点很重要：

- 如果你不执行 `git fetch`
- 你本地看到的 `origin/main` 可能已经过时

所以判断“远端有没有新提交”，最好先 `fetch` 再看。

### 0.9 `ahead` / 本地领先远端

例如：

```text
main...origin/main [ahead 2]
```

意思是：

- 你本地分支比当前记录的 `origin/main` 多 2 个提交

这通常代表：

- 你本地有提交还没推送

注意：

- `ahead` 不代表远端一定没有更新
- 只是说明按你当前本地记录来看，你本地比远端快

### 0.10 `behind` / 本地落后远端

例如：

```text
main...origin/main [behind 3]
```

意思是：

- 远端分支比你本地多 3 个提交

这通常意味着：

- 你需要同步远端最新历史

如果这时工作区还不干净，先不要直接 `pull`，应先提交或 `stash`。

### 0.11 分叉 / Diverged

分叉表示：

- 你本地有自己的提交
- 远端也有你本地没有的提交

这通常比单纯 `ahead` 或 `behind` 更复杂，因为：

- 同步时更容易出现冲突
- 更需要先整理工作区再做 `rebase` 或合并

### 0.12 `fetch`

`git fetch` 的作用是：

- 从远端拉取最新历史信息到本地
- 但不改你的工作区文件
- 也不自动合并到当前分支

它是最安全的“先看远端发生了什么”的动作。

推荐理解为：

- `fetch` = 更新情报
- `pull` = 更新情报 + 尝试整合

### 0.13 `pull`

`git pull` 本质上等于两步：

1. `git fetch`
2. 再把远端变更整合到当前分支

风险点在第二步。

如果工作区不干净，或者本地已经有很多未整理改动，`pull` 容易带来：

- 合并冲突
- 现场混乱
- 难以判断哪些改动来自你，哪些来自远端

所以本文件反复强调：工作区不干净时，不要先 `pull`。

### 0.14 `pull --rebase`

这是更推荐的同步方式。

核心逻辑是：

- 先拿到远端最新提交
- 把你的本地提交暂时拿开
- 先应用远端历史
- 再把你的提交重新放到最新历史后面

优点：

- 历史更线性
- 不容易产生无意义 merge commit
- 更适合小步修复和审查闭环场景

前提：

- 你本地改动已整理好
- 最好工作区干净

### 0.15 `merge`

合并就是把两条历史整合到一起。

普通 `git pull` 很可能触发 merge。

合并本身没有问题，但在“审查后修一个小问题”的场景下，merge 的缺点比较明显：

- 容易产生额外 merge commit
- 历史噪音更大
- 排查问题时不如 rebase 直观

### 0.16 merge commit

merge commit 是一次专门记录“把两条分支合到一起”的提交。

它不是业务改动本身，而是历史整合记录。

有些团队喜欢保留，有些团队更偏向线性历史。在“审查后修一个小问题”的场景下，通常不希望频繁出现它。

### 0.17 冲突 / Conflict

冲突表示：

- git 无法自动判断两边改动该怎么合并

典型原因：

- 你和远端同时改了同一段代码
- 你删了文件，但远端修改了这个文件
- 你改了函数签名，远端也改了附近逻辑

冲突不是错误，但代表你必须人工判断最终结果。

### 0.18 `stash`

`git stash` 是把你当前未提交的工作区和暂存区改动临时收起来。

常用场景：

- 你现在不想提交这些改动
- 但又想先切分支、拉取、测试别的东西

它适合临时避让，不适合长期堆积。

要点：

- `stash` 不是正式保存版本
- 不应把重要工作长期只放在 stash 里

### 0.19 审查 / Review

这里的“审查”指你在 Codex 或其他审查上下文里，对补丁是否正确、是否有回归、是否有 bug 做判断。

审查结果一般分三类：

- 无问题
- 有建议但非阻断
- 有 findings，需要修复

后文说“审查闭环”，就是把第三类问题真正修完并重新确认。

### 0.20 finding

finding 是审查中明确指出的一条问题。

它通常具备几个特征：

- 能定位到具体代码
- 能说清楚为什么是问题
- 原作者知道后大概率会修

不是所有评论都算 finding。只有明确、可操作、确实影响正确性的结论，才值得进入修复闭环。

### 0.21 审查闭环

审查闭环指这套完整动作：

1. 发现问题
2. 修复问题
3. 验证修复
4. 重新审查
5. 确认原问题已消除且没有引入新的阻断问题

如果只做了第 2 步，没有重新验证和复审，就不算真正闭环。

### 0.22 干净工作区

干净工作区表示：

- `git status` 不再显示未处理改动

这通常是执行下面动作之前的理想状态：

- `pull --rebase`
- 切换分支
- 推送重要提交
- 开始下一轮独立任务

### 0.23 现场

文档里提到“先处理现场”，意思是先把当前本地混杂状态整理清楚。

所谓现场，通常包括：

- 还没提交的代码改动
- 文档变更
- 提示词归档
- 报告与日志
- 测试产物

如果不先整理现场，就很难判断后续同步动作是否安全。

### 0.24 测试产物 / Generated Artifacts

测试产物是测试、构建、截图、数据库快照、报告等运行生成文件。

例如：

- `dist/`
- `test_reports/` 下某些文件
- 临时截图
- 本地构建输出

它们不一定都应该提交。

审查后最常见的问题之一就是：

- 把真正的代码修复和一堆测试产物混在一起提交

### 0.25 最小必要验证

“最小必要验证”不是随便测一下，而是：

- 只跑与当前改动直接相关、但足以证明修复有效的一组验证

例如：

- 改后端函数，就至少跑语法检查和相关单测
- 改前端页面，就至少跑构建和受影响路径验证

目标不是一次性覆盖全仓，而是在当前任务成本可控的前提下，证明补丁基本可靠。

### 0.26 修复分支

修复分支就是专门为了处理某个审查问题或缺陷而创建的分支。

推荐原因：

- 和 `main` 隔离
- 更容易做小范围提交
- 更容易回滚和复审

常见命名可以是：

- `fix/order-submit-idempotent`
- `codex/review-followup-10197`

### 0.27 同步远端

“同步远端”不是固定等于某一个命令，它可能是：

- `git fetch origin`
- `git pull --rebase`
- `git push`

要看你的目标是什么：

- 想知道远端有没有变化：`fetch`
- 想把远端变化整合到本地：`pull --rebase`
- 想把本地提交发上去：`push`

### 0.28 为什么“不要先 pull”

这是本文件最核心的习惯之一。

原因不是 `pull` 这个命令本身有问题，而是很多人在下面这些情况下会误用它：

- 刚做完审查，工作区还有一堆未整理改动
- 还没决定哪些文件该提交
- 还没修完发现的问题
- 当前就在 `main` 上直接干活

这时先 `pull`，很容易出现三类后果：

- 合并冲突提前爆炸
- 你自己的改动和远端改动混在一起
- 后续很难解释“哪部分是修 review 的，哪部分是同步带进来的”

所以更安全的顺序通常是：

1. 先看状态
2. 先整理现场
3. 先修问题并验证
4. 再同步远端

---

## 1. 先检查本地状态

先执行：

```powershell
git status --short --branch
```

目标：

- 确认当前分支
- 确认工作区是否干净
- 确认本地相对远端是 `ahead`、`behind` 还是分叉

规则：

- 工作区不干净时，不要直接 `git pull`
- 如果当前就在 `main` 上，优先考虑先切修复分支再继续

---

## 2. 判断审查结果是否需要改代码

分两类处理：

### A. 审查无阻断问题

如果 review 结论是补丁正确，通常只需要：

1. 整理当前改动
2. 提交或推送
3. 再同步远端

### B. 审查发现问题需要修复

如果 review 有 findings：

1. 先固定当前状态
2. 再进入修复
3. 修复后重新验证
4. 再做一轮审查闭环

---

## 3. 如果工作区不干净，先处理现场

### 情况 A：这些改动就是你要保留的

继续在当前分支修复，或者先提交一个中间版本。

### 情况 B：这些改动暂时不想处理

先暂存：

```powershell
git stash push
```

### 情况 C：改动很多而且混杂

先做分组，不要急着拉取远端：

- 代码修复
- 文档同步
- 提示词归档
- 日志/测试产物

---

## 4. 修复问题时的推荐顺序

1. 根据审查结论修代码
2. 补测试或更新测试
3. 跑最小必要验证
4. 再做一次审查闭环

常用验证：

```powershell
python -m py_compile <files>
python -m pytest <tests> -q
cd frontend
npm run build
```

---

## 5. 修复完成后再同步远端

不要在“未整理工作区”的情况下直接拉取。

推荐顺序：

```powershell
git fetch origin
git status --short --branch
```

然后判断：

- 如果本地只 `ahead` 不 `behind`：可以直接提交或推送
- 如果本地 `behind`：先 `git pull --rebase`
- 如果既有本地改动又有远端更新：先提交或 `stash`，再 `pull --rebase`

---

## 6. 推荐使用 `pull --rebase`

优先使用：

```powershell
git pull --rebase
```

原因：

- 历史更干净
- 不容易产生无意义 merge commit
- 更适合代码审查后的小步修复流程

前提：

- 你的工作区已经干净，或改动已被提交 / stash

---

## 7. 不建议直接在这些情况下 pull

以下情况先不要 `pull`：

- 工作区有未处理改动
- 审查刚发现问题，还没修
- 你还没决定哪些文件要提交
- 仓库里同时有提示词归档、日志、测试产物等流程文件改动

---

## 8. 最稳妥的固定流程

每次审查后都按这个顺序：

1. `git status --short --branch`
2. 判断是否需要修复
3. 如需修复，先修代码并验证
4. 再做审查闭环
5. 提交当前改动
6. `git fetch origin`
7. 如有需要，`git pull --rebase`
8. `git push`

---

## 9. 一句话原则

- 工作区不干净：先别 `pull`
- 审查有问题：先修再同步
- 准备同步远端：先提交或 `stash`
- 想保持历史干净：优先 `pull --rebase`

---

## 10. 常见状态示例

这一节专门解释 `git status --short --branch` 常见输出怎么看。你以后先看这一节，再决定要不要继续操作。

### 10.1 只有分支信息，没有文件变化

例如：

```text
## main
```

含义：

- 当前在 `main`
- 没有未提交改动
- 没有显示与远端的领先/落后关系

通常说明：

- 工作区是干净的
- 你可以安全做 `fetch`、`pull --rebase`、切分支、推送等动作

### 10.2 工作区干净，但本地领先远端

例如：

```text
## main...origin/main [ahead 2]
```

含义：

- 当前在 `main`
- 工作区干净
- 本地比 `origin/main` 多 2 个提交

通常说明：

- 你已经本地提交过，但还没推送

这时建议：

1. 先 `git fetch origin`
2. 再看远端是否也有新提交
3. 如果没有落后，直接 `git push`
4. 如果既领先又落后，再 `git pull --rebase`

### 10.3 工作区干净，但本地落后远端

例如：

```text
## main...origin/main [behind 3]
```

含义：

- 当前工作区干净
- 远端比你多 3 个提交

这时建议：

1. 直接 `git pull --rebase`
2. 如果无冲突，继续工作
3. 如果有冲突，解决后再继续

这是最适合直接同步远端的一类状态，因为没有本地未提交改动干扰。

### 10.4 本地和远端都各有提交

例如：

```text
## main...origin/main [ahead 2, behind 3]
```

含义：

- 你本地有 2 个提交没推
- 远端也有 3 个你本地还没有的提交

这时建议：

1. 确认工作区是干净的
2. 执行 `git pull --rebase`
3. 解决冲突后再推送

风险点：

- 这类状态最容易出现冲突
- 如果你此时还有未提交改动，就更不该直接 pull

### 10.5 当前有未提交修改

例如：

```text
## main
 M backend/services/tool_io_service.py
 M frontend/src/pages/tool-io/OrderCreate.vue
```

含义：

- 当前在 `main`
- 工作区不干净
- 这些文件有变动，但还没处理完

这时建议：

- 先不要 `pull`
- 先判断这些改动是不是你要保留的
- 要保留就继续修或先提交
- 暂时不处理就先 `stash`

### 10.6 既有未提交改动，又显示领先/落后

例如：

```text
## main...origin/main [ahead 1]
 M backend/services/tool_io_service.py
 M docs/API_SPEC.md
```

含义：

- 你本地已经有提交
- 但现在工作区里还有没提交的改动

这时不要只盯着 `ahead` 看。

真正的问题是：

- 你的现场没有整理完

正确动作通常是：

1. 先处理未提交改动
2. 再决定同步远端

### 10.7 `M`、`A`、`D`、`R` 分别是什么意思

在 `git status --short` 里最常见：

- `M`: modified，文件被修改
- `A`: added，新增文件
- `D`: deleted，删除文件
- `R`: renamed，文件被重命名

要点：

- 这些字母本身不代表“该不该提交”
- 只代表文件状态

判断是否要提交，要结合文件类型和任务目标看。

### 10.8 为什么提示词、日志、测试产物要特别看

在你的仓库里，除了代码文件，还经常会出现：

- `promptsRec/`
- `logs/`
- `test_reports/`
- 文档归档文件

这些改动往往不是“纯代码修复”。

所以看到下面这种状态时：

```text
M  backend/services/tool_io_service.py
A  promptsRec/archive/...
M  promptsRec/.sequence
M  test_reports/e2e_sensing.db
```

你不能简单理解成“都是同一组改动”。

更合理的做法是先分组：

- 真正的代码修复
- 流程归档文件
- 文档同步
- 测试生成物

## 11. 当前仓库案例说明

这一节直接用你这次仓库的真实状态做解释。

### 11.1 你当前看到的核心状态

本次检查中，核心输出是：

```text
## main...origin/main [ahead 2]
```

这说明：

- 当前分支是 `main`
- 按你本地当前记录，`main` 比 `origin/main` 多 2 个提交

但这还不是全部。

你同时还有一批文件改动，包括：

- 后端代码
- 前端代码
- 文档
- `promptsRec/.sequence`
- 提示词归档文件
- 测试文件
- 测试报告数据库

所以你的真实状态不是“可以马上同步”的状态，而是：

- 本地已经有提交
- 工作区里还有一批待整理改动
- 当前就在 `main`

### 11.2 这代表什么

这代表你现在最不该做的事情就是：

```powershell
git pull
```

原因不是远端一定有问题，而是你本地现场太杂：

- 代码修复和流程文件混在一起
- 有些改动应该提交
- 有些改动可能只是产物或过程文件
- 你还没把当前这批工作收口

这时先 pull，很容易把远端的新变化也卷进来，后面更难判断。

### 11.3 你现在应该优先做什么

你现在更合理的顺序是：

1. 先整理当前改动
2. 确认哪些文件属于这次修复，哪些只是过程噪音
3. 先提交或暂存当前现场
4. 再 `fetch`
5. 最后决定要不要 `pull --rebase`

### 11.4 对你当前仓库更具体一点的建议

你现在的文件可以大致按下面理解：

#### 第一组：明显属于本次修复的文件

- `backend/database/repositories/order_repository.py`
- `backend/services/tool_io_service.py`
- `frontend/src/pages/tool-io/OrderCreate.vue`
- `tests/test_order_repository.py`
- `tests/test_tool_io_service_submit.py`
- `docs/API_SPEC.md`

这组通常应该保留。

#### 第二组：流程运行产生的文件

- `promptsRec/.sequence`
- `promptsRec/archive/...`
- `logs/...`

这组是否提交，要看你们团队是否把提示词执行资产一并纳入版本控制。按你这个仓库的规则，很多情况下它们是需要保留的。

#### 第三组：需要单独判断的生成物

- `test_reports/e2e_sensing.db`
- 各类截图、调试图片

这组最需要谨慎。它们往往不是“代码修复本身”，不一定应该跟代码一起提交。

### 11.5 所以你现在该先 pull 吗

结论仍然是：

- 不该先 pull

更稳妥的顺序是：

1. 先把当前改动收口
2. 如有不想提交的产物，先剔除或暂存
3. 再执行：

```powershell
git fetch origin
git status --short --branch
```

4. 如果这时工作区干净，而且显示 `behind` 或分叉，再：

```powershell
git pull --rebase
```

### 11.6 一条适合你当前场景的简单判断句

以后你看到下面两件事同时成立：

- 当前在 `main`
- `git status` 里除了代码外，还有提示词、日志、测试产物一起变化

默认动作应该是：

- 先整理
- 先别 pull

## 12. 最终记忆版

你可以把这一段当成最短口诀：

1. 先看 `git status --short --branch`
2. 只要工作区不干净，就先别 `pull`
3. 审查有 findings，就先修再同步
4. 同步前先提交或 `stash`
5. 真要同步，优先 `git fetch`，然后 `git pull --rebase`
