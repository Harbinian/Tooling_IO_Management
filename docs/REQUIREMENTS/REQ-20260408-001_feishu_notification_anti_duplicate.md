## 5W2H 分析 / 5W2H Analysis

### What - 核心需求

**功能描述**：
所有发送飞书通知的按钮，在点击后必须立即禁用并标记为"已发送"状态，持久化到数据库，禁止重复点击刷屏。

**输入**：
- 用户点击飞书通知按钮
- 后端接口返回发送结果

**输出**：
- 按钮状态更新为"已发送"（禁用）
- 数据库记录通知发送状态
- 前端按钮 UI 反映最新状态

### Why - 动机与价值

**业务背景**：
当前系统中的飞书通知按钮可以被用户多次点击，导致同一订单的通知被重复发送到飞书群，造成信息刷屏和干扰。

**预期价值**：
- 避免重复通知导致的群消息刷屏
- 通知发送状态可追溯
- 提升系统专业性和用户体验

**成功标准**：
- 同一通知按钮在数据库记录"已发送"后，前端必须禁用
- 订单状态被驳回时，对应按钮应恢复可点击状态
- 发送失败时显示明确错误原因

### Who - 角色与用户

**主要用户**：
- 班组长（出库最终确认）
- 保管员（确认/驳回）
- 运输员（运输通知）
- 管理员（系统配置）

**开发负责人**：
- Claude Code

**受影响者**：
- 所有使用飞书通知功能的用户
- 收到重复通知的飞书群成员

### When - 时间线

**期望完成时间**：
- 2026-04-08（尽快）

**使用频率**：
- 每次订单状态流转时触发

**优先级**：
- P1（重要，当前 sprint 内完成）

### Where - 使用场景

**使用环境**：
- 生产环境
- 所有飞书通知相关页面

**集成需求**：
- 飞书 Webhook API
- 数据库 notification 状态持久化

**约束条件**：
- 按钮状态需要持久化到数据库
- 驳回操作后需恢复按钮状态

### How - 实现方式

**建议实现方式**：
1. 在 `tool_io_notification` 表增加 `is_sent` 字段
2. 或在 `tool_io_order` 表增加各步骤的通知状态字段
3. 前端按钮点击后立即禁用，后端返回成功后才更新状态
4. 如果后端返回该步骤已发送，前端保持禁用

**参考系统**：
- 现有 `keeperConfirmPayload.js` 中的状态管理逻辑

**技术栈**：
- 后端：Flask + SQL Server
- 前端：Vue 3 + Element Plus

### How Much - 资源与范围

**范围**：
- [ ] 包含：
  - 数据库增加通知发送状态字段
  - 后端 API 返回通知状态
  - 前端按钮禁用逻辑
  - 驳回后恢复机制

- [ ] 不包含：
  - 飞书 API 本身的重试逻辑（已有）
  - 其他通知渠道

**资源需求**：
