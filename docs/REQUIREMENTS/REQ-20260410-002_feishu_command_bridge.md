## 5W2H 分析 / 5W2H Analysis

### What - 核心需求

**功能描述**：
实现飞书消息与 Claude Code CLI 之间的双向桥接：
1. 后台 HTTP 服务接收飞书消息（`/claude` 前缀），写入共享队列
2. Claude Code CLI 轮询队列，取消息执行；执行中进度写入执行状态文件，最终结果写入结果文件
3. 后台服务检测执行状态与结果文件，推送飞书进度、确认提示和最终回复

**输入**：
- 飞书消息（`/claude` 开头）
- 用户在 CLI 中的确认操作

**输出**：
- 飞书文本回复（执行进度、执行结果或确认提示）

---

### Why - 动机与价值

**业务背景**：
手机开发场景下，需要通过飞书远程触发 Claude Code 执行工作，并获取执行结果。

**预期价值**：
- 摆脱设备限制，随时通过手机驱动开发
- 异步执行，不阻塞手机操作
- 统一入口，通过飞书触发各种技能和操作

**成功标准**：
- 飞书发送 `/claude xxx` 后，CLI 能收到并执行
- 执行过程中能看到任务级别进度推送
- 执行结果能推送到飞书

### How - 实现方式

**架构方案**：
```
┌─────────┐  webhook   ┌──────────────┐  写入   ┌─────────┐
│ 手机飞书│ ─────────→  │ 后台HTTP服务 │ ──────→ │ 消息队列 │
└─────────┘            │  (端口8152)  │         │  文件   │
                        └──────────────┘         └────┬────┘
                                                       │ 轮询
                        ┌──────────────┐  写入   ┌────▼────┐
                        │ 后台HTTP服务 │ ←─────  │Claude CLI│
                        │ (结果推送)   │  结果   └────┬────┘
                        └──────┬───────┘  文件         │
                               │ 推送                   │
                               ↓                       │
                        ┌──────────────┐               │
                        │ 手机飞书     │ ←──────────────┘
                        └──────────────┘
```

**组件清单**：

| 组件 | 文件 | 职责 |
|------|------|------|
| 后台 HTTP 服务 | `scripts/feishu_bridge/server.py` | 接收飞书消息、推送结果 |
| 消息队列 | `scripts/feishu_bridge/queue.py` | 消息读写、锁机制 |
| CLI 轮询入口 | `scripts/feishu_bridge/cli.py` | 轮询队列、执行指令、结果回写 |
| 指令解析器 | `scripts/feishu_bridge/parser.py` | 解析 `/claude` 指令 |

**交互流程**：
1. 用户飞书发送 `/claude scan gates G1 G2`
2. 后台服务写入队列 `queue/pending/{uuid}.json`
3. CLI 轮询发现新消息，将其移入执行中状态文件
4. CLI 执行过程中持续写入任务级别进度
5. 后台服务检测到新进度，推送飞书
6. 如需确认，CLI 在本地终端提示操作者确认，后台服务推送“等待本地确认”提示
7. CLI 执行完成后写结果到 `queue/results/{uuid}.json`
8. 后台服务检测到结果，推送飞书

**指令格式**：
- `/claude <text>` - 执行任意指令
- `/claude /skill-name <args>` - 调用其他技能
- `/claude status` - 查看服务状态
- `/claude help` - 显示帮助
