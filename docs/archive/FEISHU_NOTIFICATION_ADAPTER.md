# Feishu Notification Adapter

## Adapter Architecture

The Feishu adapter adds an external delivery layer on top of the internal notification framework.

Primary module:

- `backend/services/feishu_notification_adapter.py`

Responsibilities:

1. map normalized notification payloads to Feishu webhook text messages
2. load delivery configuration from `config/settings.py`
3. create a dedicated `feishu`-channel notification record for each delivery attempt
4. send the outbound request
5. persist delivery status and response summary back into the notification record

The internal notification record remains the business source of truth. Feishu delivery uses a separate notification row so internal read/unread state does not conflict with external send status.

## Configuration Loading Rules

The adapter reads configuration through `config/settings.py`.

Supported settings:

| Setting | Purpose |
|---|---|
| `FEISHU_NOTIFICATION_ENABLED` | global enable / disable flag |
| `FEISHU_NOTIFICATION_TIMEOUT_SECONDS` | outbound request timeout |
| `FEISHU_WEBHOOK_URL` | default webhook |
| `FEISHU_WEBHOOK_TRANSPORT` | transport-specific webhook override |

Behavior:

- if delivery is disabled, the adapter returns a non-blocking `disabled` result
- if webhook configuration is missing, the adapter records a failure result
- secrets are never hardcoded in source files

## Message Mapping Strategy

The adapter reuses the normalized notification payload generated by `notification_service`.

Outbound message content includes:

- notification title
- order number
- notification type
- receiver marker when available
- business body text

Current auto-delivery event types:

- `KEEPER_CONFIRM_REQUIRED`
- `TRANSPORT_REQUIRED`
- `ORDER_COMPLETED`

Other notification types are skipped unless explicitly sent through a manual service path.

## Delivery Result Persistence

For each Feishu send attempt, the adapter:

1. creates a `feishu`-channel notification record
2. sends the webhook request
3. updates that record with:
   - `send_status`
   - `send_time`
   - `send_result`

This preserves UI-readable delivery information without mutating the internal record used for inbox behavior.

## Failure Handling Rules

Feishu delivery is best-effort.

- order workflow state changes must still succeed if Feishu delivery fails
- failures are logged without leaking webhook values
- the adapter returns normalized failure summaries for future retry tooling
- unsupported event types are skipped cleanly

## Workflow Trigger Points

Current integration points in `backend/services/tool_io_service.py`:

- internal notification creation path now attempts Feishu auto-delivery for selected stable events
- existing manual send flows for keeper and transport notifications now route through the adapter

- configuration-driven disabled path

The adapter already exposes a retry-oriented helper pattern and returns normalized delivery results.

- richer Feishu card message templates
- per-event routing configuration
