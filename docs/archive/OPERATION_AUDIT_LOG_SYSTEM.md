# Operation Audit Log System

## Purpose

The Tool IO workflow writes one audit record for every key business transition so the system can answer:

- who performed the operation
- what action was executed
- which order was affected
- when the action happened
- which status changed from and to
- what remark or business note was attached

## Audit Record Fields

The current implementation reuses the existing Tool IO operation log table. Each record is written with these logical fields:

| Field | Source |
|---|---|
| `order_id` | stored as `order_no` |
| `operation_type` | stable operation code such as `create`, `submit`, `keeper_confirm`, `transport_notify`, `final_confirm`, `cancel`, `reject` |
| `operator_user_id` | authenticated user id from route payload |
| `operator_name` | authenticated display name |
| `operator_role` | route role fallback or supplied operator role |
| `operation_time` | generated by the database on insert |
| `previous_status` | status before the successful transition |
| `new_status` | status after the successful transition |
| `remark` | business note, failure note, or user remark |

## Write Path

Unified write entry:

- `backend/services/audit_log_service.py`

Primary API:

- `write_order_audit_log(...)`

Behavior:

- normalizes the audit payload shape
- writes into the existing operation log table through `add_tool_io_log`
- catches all exceptions
- logs server-side errors without breaking the main workflow

## Integration Points

Current integration points in `backend/services/tool_io_service.py`:

- order creation
- order submission
- keeper confirmation
- final confirmation
- order rejection
- order cancellation
- transport notification success/failure
- keeper notification send result

Runtime keeper processing in `backend/services/tool_io_runtime.py` now returns status transition metadata to the service layer instead of writing logs directly. This keeps the final audit write centralized.

Database transition functions in `backend/database/tool_io_queries.py` also return transition metadata and no longer emit duplicate audit rows for the main lifecycle operations.

## Failure Handling

Audit logging is best-effort by design.

- A failed audit insert must not fail the main order transaction.
- Failures are written to application logs through the Python logger.
- This protects production flows from audit-table connectivity or schema issues.

## Extension Guidance

Future extensions should follow the same pattern:

1. perform the business transition first
2. return or compute `previous_status` and `new_status`
3. call `write_order_audit_log(...)` after success
4. keep audit failures non-blocking

Recommended next extensions:

- dedicated `transport_start` audit event if a separate transport execution step is introduced
- system correction audit events for admin repair tools
- normalized audit response mapping for frontend log presentation
- filtering and reporting APIs for auditor workflows
